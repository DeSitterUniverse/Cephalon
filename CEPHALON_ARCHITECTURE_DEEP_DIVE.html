Cephalon Architecture Deep Dive

Current architecture for the local-first desktop RAG workbench. This document describes how the app is built now, not a release checklist.

System Overview

Tauri desktop shell
  -> React workbench
      -> typed API client
      -> SSE live events
  -> Python sidecar or dev backend
      -> FastAPI routes
      -> SQLite metadata / FTS5
      -> LanceDB dense vectors
      -> ONNX Runtime embedder + reranker
      -> llama.cpp GGUF generation

Desktop Shell

src-tauri owns the native window, Tauri permissions, file dialogs, sidecar launch, and packaging. The shell keeps frontend assets and backend launch mechanics separate from model/data folders.

Frontend

src is a React/Vite app. TanStack Query caches backend state. Zustand stores UI-only selections such as active model, panels, theme, selected sources, and event-stream status.

Backend

python/cephalon_core is a FastAPI package split into routes, config, schemas, storage, ingestion, retrieval, generation, jobs, metrics, events, observability, evaluation, support, and model setup services.

Storage Responsibilities

SQLite

SQLite is the source of truth for metadata and observability: documents, tags, jobs, job events, settings, conversations, messages, message sources, parent chunks, summary nodes, child chunks, retrieval traces, eval runs, answer records, citations, feedback, and FTS5 rows.

LanceDB

LanceDB stores dense vectors only. It does not own document metadata, lexical search, or hybrid fusion. This keeps vector storage simple and avoids depending on LanceDB's Tantivy-backed hybrid path.

The installer packages the app and backend runtime. Embedder/reranker artifacts are installed into the user model directory from Settings by using configured prepared ONNX repos or selecting exported local folders. A valid ONNX engine folder contains model.onnx, tokenizer files, onnx_profile.json, and any external ONNX data files referenced by the model.

Ingestion Pipeline

1. Queue. Files, folders, and the configured Obsidian vault enter the durable job queue. The UI receives job/document updates through SSE.
2. Extract. Known formats use format-specific parsers. Unknown text-like files are imported with encoding sniffing and binary guards. Extraction warnings and failures are stored instead of disappearing.
3. Fingerprint. Documents and chunks receive content hashes, parser version, chunking profile, embedding model id, embedding dimension, and timestamps for stale detection.
4. Chunk. The backend creates parent chunks for generation context, smaller child chunks for precise retrieval, and summary nodes for high-level steering.
5. Index. Child chunks are written to SQLite, SQLite FTS5, and LanceDB. Reindex/delete operations clean SQLite rows and LanceDB vectors for the affected document.

Retrieval Pipeline

question
  -> normalize and optionally split into subqueries
  -> embed query
  -> summary steering
  -> dense child search in LanceDB
  -> lexical BM25 search in SQLite FTS5
  -> reciprocal rank fusion
  -> Jina reranker
  -> parent context reconstruction
  -> context compression
  -> typed source/meta/token stream
  -> retrieval trace persistence

Hybrid retrieval is explicit: vector and lexical searches run independently, then merge by reciprocal rank fusion. This makes every score visible and keeps BM25 behavior local, transactional, and testable.

Observability

Retrieval Trace

Each saved trace records raw query, normalized query, subqueries, vector candidates, BM25 candidates, fused candidates, reranked candidates, final context, unused candidates, score fields, latency, and no-answer gate values.

Index Health

Index health summarizes document/chunk counts, embedded chunks, stale state, failed ingestions, parse warnings, duplicate chunks, chunk length stats, never-retrieved documents, retrieval counts, index size, embedding distribution, and chunking profiles.

Answer Support

Answer records and citations are stored separately from chat messages. Support labels are deterministic: supported, weak, or unsupported based on final context presence and retrieval/rerank scores.

Build And Packaging

The development path uses the Python executable on PATH. The release path builds the frontend, checks Rust, builds a PyInstaller sidecar, and packages Tauri. The sidecar includes backend code and runtime libraries but intentionally excludes ONNX model artifacts. This keeps the installer smaller and lets model engines be updated separately from app releases.

Failure Handling

  • Missing ONNX engines create a visible setup state; the app does not silently fall back to another embedder or reranker.
  • Querying without a loaded GGUF returns a conflict response and the UI keeps the run action disabled.
  • Model diagnostics report Python runtime, llama.cpp package path, backend libraries, active context, and load errors.
  • Metrics and trace write failures are non-fatal where possible and surfaced through health or diagnostics.
  • Event-stream disconnects trigger reconnect attempts and temporary polling rather than permanently lying about state.
