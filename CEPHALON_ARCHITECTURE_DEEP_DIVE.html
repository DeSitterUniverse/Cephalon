Cephalon Architecture Deep Dive

Current architecture for the local-first desktop RAG workbench.

System Shape

Desktop Shell

src-tauri owns the native window, sidecar launch, file dialog permissions, packaging resources, and release-time configuration. The app uses a custom titlebar so the desktop frame and workbench feel like one surface.

Frontend

src is a React/Vite app. TanStack Query handles server state, Zustand stores local UI state, and the typed API client centralizes backend calls and query streaming.

Backend

python/cephalon_core is the FastAPI backend package. The package is split by responsibility: routes, storage, ingestion, retrieval, generation, models, jobs, documents, events, metrics, schemas, and config.

Observability Data

Cephalon records retrieval diagnostics in SQLite so the user can debug why an answer used specific evidence. The trace tables store the query, subqueries, stage candidates, final context, unused candidates, score fields, no-answer diagnostics, and latency. Answer records store confidence and citation support separately from chat messages.

Ingestion Pipeline

1. Resolve input. A queued job receives a file, folder, or configured Obsidian vault path. Supported file types use native extraction. Unknown text-like files are imported as text after binary guards. Obsidian imports skip .obsidian and other internal folders.
2. Register document. SQLite records path, content hash, extraction mode, indexing state, display name, size, timestamps, and embedding metadata.
3. Build hierarchy. Text is split into semantic units, grouped into parent chunks, summarized into summary nodes, and broken into smaller child chunks for precise retrieval.
4. Index. Child chunks go to SQLite, SQLite FTS5, and LanceDB. Summary nodes go to SQLite and LanceDB. Reindexing clears generated rows and vectors for the document before writing fresh state.

Retrieval Pipeline

1. Query planning. Compound prompts are split into bounded subqueries. Exact numeric max/min/sort questions can use a deterministic indexed-row scan before generation.
2. Summary steering. Summary vectors identify likely parent regions and provide a parent-level boost.
3. Hybrid search. Dense child search runs in LanceDB. Lexical BM25 search runs in SQLite FTS5. Results are merged with reciprocal rank fusion.
4. Reranking. Jina reranker ONNX scores fused candidates. Retrieval priors keep strong lexical matches from being displaced by unrelated dense matches.
5. Context reconstruction. Child hits point back to parent chunks. The prompt receives compressed parent context with stable source tags such as [[src:S1]].
6. Streaming. The backend streams typed SSE events for subqueries, conversation ids, source metadata, answer metadata, tokens, saved message ids, errors, and completion.
7. Trace persistence. Candidate lists, scores, ranks, no-answer diagnostics, latency, and final context are written to SQLite when trace persistence is enabled.

Index Health

The index health API summarizes document count, chunk count, embedded chunks, stale documents, failed ingestions, parse warnings, duplicate chunk rate, chunk length distribution, never-retrieved documents, retrieval counts, index size, embedding model distribution, and chunking profile distribution. These values are computed from SQLite and the LanceDB data directory; they do not require cloud services or a separate analytics database.

Evaluation

The eval harness accepts small JSON eval sets and records deterministic retrieval metrics. The first metrics are Recall@k, MRR, expected document hit rate, and expected chunk hit rate. This keeps quality checks cheap enough to run locally after changing chunking or retrieval settings.

Model Runtime

Embedding and reranking use ONNX Runtime to avoid a PyTorch runtime requirement in the normal app path. The embedder is jinaai/jina-embeddings-v5-text-small and the reranker is jinaai/jina-reranker-v3. llama.cpp loads the selected GGUF chat model only after the user presses Load. The backend reports llama.cpp library paths, backend availability, active context size, and model metadata through health/model APIs.

Chat History And Memory

Chats are stored in SQLite as conversations, messages, and per-message source metadata. This gives the app durable history and allows later retrieval over summarized conversation memory. This is retrieval memory, not training: model weights do not change.

Frontend Data Flow

The frontend uses TanStack Query for health, models, documents, jobs, settings, conversations, and document details. Zustand stores selected model, selected document, selected conversation, selected sources, right panel, and live event state. Query streaming is consumed as typed events so sources can arrive before generation tokens.

Failure Handling

  • Missing or invalid ONNX artifacts create a structured startup error instead of silent fallback.
  • Querying without a loaded GGUF returns a conflict response and the UI keeps the Run button disabled.
  • Unsupported/binary ingestion failures are stored as failed document or job state with a clear reason.
  • Metrics and trace write failures are non-fatal where possible and surfaced through health or diagnostics.
  • SQLite FTS5 search is explicit and testable, avoiding LanceDB Tantivy Python-package drift.
